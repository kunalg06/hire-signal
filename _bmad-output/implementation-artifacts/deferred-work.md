# Deferred Work

## Deferred from: code review of 6-5-guarded-mode-claude-restrictions (2026-07-03)

- **Guarded-mode injection failure is silent** (`app/services/docker_service.py`, `inject_workspace_files()`): found during the re-review after the chmod-skip patch. If the `CLAUDE.md` write itself fails (transient Docker/disk issue — distinct from a candidate deliberately deleting the file post-injection), the function logs a debug-level warning and link generation proceeds normally. No employer or candidate is told the assessment silently ran unguarded. Future story: surface `ai_assistance_mode` and injection success/failure on the teacher dashboard / submission record so a failed guarded-mode injection is visible after the fact.

- **Guarded mode is enforcement-by-honor-system, trivially bypassable** (`app/services/docker_service.py`, `student_preview`... n/a — `inject_workspace_files()`): The entire mechanism is a `CLAUDE.md` file politely asking the in-container Claude Code CLI to restrict itself. The candidate has a full shell in the container and `/workspace` is world-writable (`drwxrwxrwx`, confirmed live) — they can delete/edit the file, invoke `claude` from a different working directory so it never loads, or use an AI tool entirely outside the sandbox. Decided during code review (2026-07-03, user choice): ship as-is for v1 as a soft nudge, not a hard guardrail. Real backend enforcement (e.g. proxying/validating Claude API calls from the container, an immutable bind-mount for CLAUDE.md) is a future story if assessment integrity requirements demand it.
- **No whitelist/validation on `ai_assistance_mode` string match** (`app/routes/links.py:29`): Comparison is an exact string match against `'guarded'`; any drift (typo, case, whitespace from a future manual DB edit or code change) silently falls back to unrestricted with no warning. Low practical risk today since the value is enum-validated at challenge creation (`POST /api/generate-challenge` / `/api/challenges` reject anything outside `{guarded, unguarded}`), and hardening the string match wouldn't meaningfully improve security anyway given the honor-system limitation above.
- **Magic string `'unguarded'` duplicated as a default value** (`app/routes/links.py:26` and `app/services/docker_service.py:107`): Two independent hardcoded literals that must stay in sync by convention. Future story: extract to a shared constant if a third mode is ever added.
- **No try/except around the new `db_service.get_challenge()` call** (`app/routes/links.py:30`): If it raises (DB lock/connection issue), the whole `generate_link` request fails. Matches the existing convention two lines above (`get_assignment()` is equally unguarded) — not a new fragility introduced by this story, just extending the existing pattern.
- **AC3's "(deleted/soft-deleted challenge)" example is currently unreachable**: `get_challenge()` fetches "regardless of publish status" (its own docstring) and the app has no hard-delete path for challenges — `DELETE /api/challenges/<id>` only sets `is_published = -1`, the row and its `ai_assistance_mode` persist forever. So a soft-deleted guarded challenge keeps enforcing guarded mode on any assignment still referencing it — arguably the *correct* behavior (an already-created assignment shouldn't silently lose its restriction because the catalog entry was unpublished), but it means AC3's stated fallback path never actually triggers in practice. Spec-wording note for a future story-file cleanup, not a code defect — the code correctly implements AC3's literal text (falls back to `'unguarded'` only when `get_challenge()` returns `None`).
- **`chmod 666` on the injected `CLAUDE.md`** (`app/services/docker_service.py:206`): Matches the existing permission treatment of `instructions.md`/`solution.py`, but means the candidate has write access to their own restriction file — component of the honor-system decision above, not independently worth a different chmod value (since `/workspace` itself is world-writable, restricting the file alone wouldn't prevent deletion).

## Deferred from: code review of 6-4-preview-as-student (2026-07-03)

- **No auth/role check on `/student/preview/<challenge_id>`** (`app/routes/student.py:647`): Any caller who knows or guesses a `challenge_id` can view its title, description, rubric, and starter code, published or not. Deferred — matches the app-wide no-auth dev posture documented in CLAUDE.md; becomes a real issue when auth is added.

- **Permanent HTTP errors (404/500) during polling treated as transient** (`app/routes/student.py:614`, `startPolling`): The `catch` block treats any fetch failure — network hiccup or permanent server error — the same way, silently retrying until the 60s timeout. A genuine backend error surfaces to the candidate as "taking longer than expected" instead of an accurate failure state. Future story: distinguish permanent (4xx/5xx) from transient errors and show a distinct error state.

- **No re-entrancy guard on `startPolling`** (`app/routes/student.py:605`): Nothing prevents a double-click or duplicate submit event from invoking `startPolling` twice concurrently, spinning up two overlapping polling loops hitting `/api/submission/<id>` in parallel. Low real-world impact given the confirm-modal step already reduces double-submit likelihood.

- **CSS duplication between `student_dashboard()` and `student_preview()` templates** (`app/routes/student.py`): Gradient headers, card styling, topbar, and buttons are hand-duplicated across both f-string templates rather than shared. Any future visual fix to the real assessment UI must be manually re-applied to the preview template or the two will drift apart.

- **Fixed poll interval/timeout, no backoff or jitter** (`app/routes/student.py:609-610`): `POLL_MS = 3000`, `TIMEOUT = 60000` are hardcoded with no jitter — every submitting candidate polls on the same fixed cadence, a needless synchronized load pattern if many candidates submit around the same time. Not a concern at current scale.

- **`escHtml` doesn't escape single quotes, unlike Python-side `html.escape`** (`app/routes/student.py:558`): Two inconsistent, hand-maintained escaping implementations for the same "render untrusted text safely" concern. Low risk today since `escHtml`'s only call site renders into a `<p>` element body, not an attribute value, but invites a future call site to assume more coverage than exists.

- **Assignment/link-keyed preview (true candidate fidelity)** (new story, unscheduled): `student_preview()` reads from the `challenges` catalog template, not the live `assignments` row a candidate actually sees — these can diverge once an employer edits an assignment post-generation. Resolved via BMad party-mode panel (John/Winston/Amelia/Sally, unanimous 2026-07-03): no frontend button exists yet for the current route, so no code change was forced now; AC3 was reworded to describe template-preview accurately instead. Follow-up story needed once UI placement is decided (assignment-detail-page trigger vs. catalog trigger) — that decision determines whether `challenge_id` uniquely resolves to one assignment (it does not; a challenge can spawn multiple assignments) and whether new disambiguation logic is required.

## Deferred from: code review of Epic 4 stories 4.1–4.5 (2026-07-02)

- **N+1 DB queries in candidates endpoint** (`app/routes/challenges.py:215`): One `get_dimension_scores()` query per candidate, unbounded. For large cohorts (100+ candidates) this will be slow. Fix: batch-load all dimension scores in one `WHERE submission_id IN (...)` query before building the candidates list. Deferred because Story 4.2 spec accepted this pattern and no pagination AC exists.

- **Flagged candidates unfiltered in candidates ranking** (`app/services/database_service.py` — `get_candidates_for_challenge` query): Flagged submissions appear in the ranked candidate list with no `is_flagged` indicator in the payload. Recruiters cannot distinguish flagged-for-review candidates from clean ones. Future story: add `s.is_flagged` to `get_candidates_for_challenge` SELECT and expose in candidate object.

- **Re-flagging overwrites prior flag metadata** (`app/routes/submissions.py:361`, `app/services/database_service.py`): Second `POST /submissions/<id>/flag` call overwrites `flag_by`, `flag_reason`, `flagged_at` with no history. Unlike overrides (which have `score_overrides` append-only log), flags have no audit trail. Future story: add `flag_events` append-only table; or add `WHERE is_flagged = 0` guard to prevent overwrite.

- **`flagged_by` is caller-supplied with no authentication** (`app/routes/submissions.py:360`): Any caller can claim any identity in `flagged_by`. Deferred — system has no auth by design (CLAUDE.md). Becomes a real issue when auth is added.

- **Concurrent overrides double-count in analytics** (`app/routes/submissions.py:371–410`): Two simultaneous POST requests to `/submissions/<id>/override` both pass the existence check and both insert into `score_overrides`, double-counting one override event. Deferred — single-tenant dev-phase system; no concurrency requirement exists.

## Deferred from: code review of story 1-4-replace-print-with-proper-logging (2026-07-02)

- **Dead `_run()` call in `get_file_from_container`** (`app/services/docker_service.py:73`): First `_run(['cp', container_id:path, '-'])` at line 73 uses `text=True` which causes `UnicodeDecodeError` on binary tar content; the working binary-mode `subprocess.run()` at line 79 is unreachable for plain files. For ASCII Python files the extra call is merely wasteful; for non-UTF-8 content the method silently returns `None`. Fix: remove the dead `_run()` call at line 73–74; the binary-mode call at line 75–79 is the one that actually works.

- **`flask run` entry point produces no INFO-level logging** (`run.py`): `logging.basicConfig()` is only configured in `run.py`. If a developer starts the server with `flask run` (which calls `create_app()` directly), the root logger defaults to WARNING level and all `logger.info()` / `logger.debug()` calls (evaluation completion, container start, file reads) are silently dropped. Fix: add a guard-style `logging.basicConfig()` call in `create_app()` — e.g., `if not logging.root.handlers: logging.basicConfig(...)` — without violating AC4's spirit of not having unconditional logging config inside `app/`.

- **Background evaluation thread does not catch `BaseException`** (`app/routes/submissions.py:17`, `evaluate_submission_files`): The outer `except Exception as e:` block does not catch `BaseException` subclasses (e.g., `MemoryError`, signal delivery). If such an exception escapes the LLM call, the thread dies silently, `update_submission_evaluation()` is never called, and the submission row stays permanently pending (NULL score/feedback) with no log entry. Deferred — very rare failure mode; no timeout/watchdog story exists yet.
